# WebHTV 远程托管能力实施设计

## 前提说明

**场景前提：被控端优先按电视设备优化**
- 电视长期在线、永久供电，3-5 秒轮询可接受。
- App 也可以运行在手机、平板、盒子上，同一套协议不固定“控制版/被控版”。
- 远程操作频率低，首要目标是简单、稳定、少打扰。
- 同一个 serverless 域名/origin 不变时，不应要求用户重新绑定；只有用户删除服务后域名变化，或主动更换到另一个 origin 时才重新绑定。

**核心设计原则：**
- 长期绑定状态保存在 App/主控端本地，不依赖 serverless 内存。
- serverless 只保存可重建的在线路由、命令队列、同步临时文件。
- 默认零配置，不要求 KV、R2、数据库或必填环境变量。
- 使用高熵 `deviceToken/groupToken` 作为 bearer capability token；`JWT_SECRET` 只作为可选增强，不是简易版必填项。
- 设备对等架构：任意 App 可主控、可被控、可作为来源设备、可作为目标设备。

## 1. 定位和术语

这个能力不绑定任何具体人际关系。无论是多台自有设备、长期托管，还是临时协助，本质都是设备之间的托管关系。产品和代码里统一使用下面的通用术语：

- `本机`：当前正在操作的 WebHTV App。
- `远端设备`：同一个设备组里的另一台 WebHTV App。
- `主控端`：发起管理操作的一端，可以是 Web 控制台，也可以是 WebHTV App。
- `被控端`：接收远程命令并执行的 WebHTV App。
- `来源设备`：一键同步的数据提供方，通常是已经配置好的 WebHTV App。
- `目标设备`：一键同步的数据恢复方。
- `设备组`：一组互相信任的 Web 控制台和 WebHTV App。一个设备组里可以有多个主控端、多个来源设备、多个目标设备。

同一个 WebHTV App 必须同时支持多种身份：

- 可以作为主控端管理别的设备。
- 可以作为被控端接收远程命令。
- 可以作为来源设备把本机配置同步出去。
- 可以作为目标设备恢复别的设备同步过来的数据。
- 可以加入多个设备组，支持“多台自有设备”“临时协助”“长期托管”等场景。

用户可见名称建议用“远程托管”或“设备托管”，不要使用绑定具体人际关系的功能名。

## 2. 类似项目参考结论

这次参考的是产品模式和架构模式，不照搬远程桌面。参考对象：

- Syncthing：`https://docs.syncthing.net/intro/getting-started.html`
- RustDesk 自建服务：`https://rustdesk.com/docs/en/self-host/`
- KDE Connect：`https://userbase.kde.org/KDEConnect`
- Home Assistant 服务动作：`https://developers.home-assistant.io/docs/dev_101_services/`

- Syncthing 使用 `local device / remote device` 术语，设备之间通过设备 ID 建立关系，数据按 folder 选择同步。对 WebHTV 的启发：不要预设人际关系，使用”本机/远端设备/来源设备/目标设备”；同步时让用户选择数据项，不让用户理解文件路径和 Cookie 细节。
- RustDesk 自建服务端分 ID/rendezvous 和 relay，客户端持续 ping ID server，需要连接时由服务端撮合或中转；同时提供 OSS 简单服务端和 Pro Web 控制台。对 WebHTV 的启发：App 主动出站连接 serverless/Go 服务端，Web 控制台只发命令，不暴露 App 本地 `/m` 到公网；简易版和完整版共用协议，只是能力不同。
- KDE Connect 是设备对设备配对，所有设备都运行同一套 App，通过插件能力决定能做什么。对 WebHTV 的启发：不要把 App 分成固定”控制版/被控版”，每台 WebHTV App 都注册自己的 capabilities。
- Home Assistant 的 Web/API 模式是”控制台调用服务动作”。对 WebHTV 的启发：远程命令应是结构化 command，例如 `config.use`、`remoteSync.restore`、`proxy.save`，不要做任意 HTTP 代理。
- **Home Assistant 使用长期访问令牌（Long-Lived Access Tokens）**，这是智能家居设备的常见做法。对 WebHTV 的启发：电视场景下长期 bearer 凭证是合理选择，不需要频繁刷新。

落到 WebHTV：采用 `设备组 + RemoteAgent + command schema + 一键同步传输 + capabilities`。serverless 只做在线中转；完整版提供持久队列、WebSocket、备份中心、文件管理和更多 App 能力。

`RemoteStore` 不进 Room。它必须是“外部文件为主、Prefers 镜像为辅”的两层存储，因为当前 `AndroidManifest.xml` 明确 `android:allowBackup="false"`，卸载 App 会清空内部数据；远程托管关系不能因此丢失。

推荐文件位置：

```text
Path.root("WebHTV/RemoteTrust/profile.json")
```

不要放在 `TV/TVBox/TVData` 下面。现有 `Setting.getSyncPaths()` 默认会同步 `TV\nTVBox\nTVData`，如果把远程托管令牌放进去，一键同步会把 `deviceToken/groupToken` 复制到另一台设备，造成设备身份克隆。

`RemoteStore` 落地规则：

- 外部文件是主存储；`Prefers` 只做启动前的快速镜像和无文件权限时的临时兜底。
- App 启动时先读外部文件；读到有效数据后覆盖 Prefers 镜像。
- 读写外部文件前用现有 `Setting.hasFileAccess()` 判断；没有文件权限时先用 Prefers 镜像运行，但设置页显示“需要文件权限以便卸载后保留托管关系”。
- `RemoteTrustDialog` 打开或保存 profile 时复用现有 `PermissionUtil.requestFile()`，不要新增权限申请流程。
- 用户授予文件权限后，立即把 Prefers 镜像写回外部文件。
- 写文件使用 `profile.json.tmp` 原子写入，成功后 rename 到 `profile.json`。
- 用户在“远程托管”里执行“移除此服务/清空托管关系”时，同时删除外部文件和 Prefers 镜像。
- 一键同步、备份、管理页同步默认不能包含远程托管令牌；需要在 `Backup.include()` 里显式排除 `remote_trust_` 前缀，并在 `SyncFiles.skip()` 里排除 `WebHTV/RemoteTrust/`。
- 只有未来单独做“导出/导入托管关系”时，才允许用户明确搬运这些令牌；普通一键同步不能顺带复制。

对应代码约束：

```java
// Backup.include(...)
if (key.startsWith("remote_trust_")) return false;

// SyncFiles.skip(...)
if (path.equals("WebHTV/RemoteTrust") || path.startsWith("WebHTV/RemoteTrust/")) return true;
```

核心模型：

```java
public class RemoteStoreFile {
    public int version = 1;
    public List<RemoteProfile> profiles;
}

public class RemoteProfile {
    public boolean enabled;
    public boolean keepOnline;
    public String serverUrl;        // 用户输入值，用于展示
    public String serverOrigin;     // 绑定边界：scheme + host + port
    public String deviceId;         // 由 serverOrigin + deviceToken 派生
    public String deviceToken;      // 高熵 bearer token，长期有效
    public List<RemoteBindGrant> pendingBindGrants;
    public List<RemoteGroup> groups;
    public long updatedAt;
}

public class RemoteGroup {
    public String groupId;
    public String groupToken;       // 高熵 bearer token，长期有效
    public String groupTokenHash;   // sha256(serverOrigin + ":" + groupToken)，本地执行命令前校验
    public String name;
    public List<RemoteDevice> devices;
    public long updatedAt;
}

public class RemoteBindGrant {
    public String grantId;
    public String bindGrantToken;   // 内部绑定恢复 token，不展示给用户
    public long createdAt;
    public long consumedAt;
}

public class RemoteDevice {
    public String deviceId;
    public String name;
    public int type;
    public String appVersion;
    public long lastSeen;
    public boolean online;
    public RemoteCapabilities capabilities;
}
```

**重要：长期有效令牌的边界**

首版不引入账号系统，也不要求 serverless 配置 secret。`deviceToken/groupToken` 是高熵随机 bearer token，由 App/主控端按 `serverOrigin` 持久保存。`serverOrigin` 的标准化规则：

1. 解析用户输入的 server URL。
2. 只保留 `scheme://host[:port]`。
3. 去掉 path、query、fragment 和结尾斜杠。
4. token 派生、profile 查找、是否需要重新绑定都只看 `serverOrigin`。

规则：

- 同一个 `serverOrigin` 不变时，实例重启、冷启动、平台回收后都继续使用原 token，不重新绑定。
- 用户删除 serverless 且域名变化，或主动换 server URL 到另一个 origin 时，App 视为新 profile，需要重新绑定。
- token 不需要服务端持久存储；serverless 重启后由设备和主控端重新上报本地保存的 token，重建在线路由表。
- `JWT_SECRET` 可以作为高级部署的可选增强，用于签名 token，但不是简易版默认要求。
- App 卸载重装后，只要 `WebHTV/RemoteTrust/profile.json` 还在，并重新授予文件权限，就恢复原 `deviceToken/groupToken`，不重新绑定。

设备身份不要只存一个”被控/主控”枚举。角色来自 capabilities 和当前操作上下文：某台设备被选为 `sourceDeviceId` 时就是来源设备，被选为 `targetDeviceId` 时就是目标设备。

## 5. RemoteAgent 生命周期

`App.startBackgroundServices()` 当前会启动：

```java
Server.get().start();
PlaybackRemoteSyncer.start();
NsdDeviceDiscovery.register();
```

建议追加（统一单例模式）：

```java
RemoteAgent.get().start();
```

`RemoteAgent` 设计：

```java
public class RemoteAgent {
    private static volatile RemoteAgent instance;

    public static RemoteAgent get() {
        if (instance == null) {
            synchronized (RemoteAgent.class) {
                if (instance == null) {
                    instance = new RemoteAgent();
                }
            }
        }
        return instance;
    }

    public void start() {
        RemoteStoreFile store = RemoteStore.get();
        boolean keepOnline = false;
        for (RemoteProfile profile : store.profiles) {
            if (!shouldStart(profile)) continue;
            startSession(profile); // capabilities -> register/resume -> poll
            keepOnline |= profile.keepOnline;
        }
        if (keepOnline) RemoteAgentService.start(App.get());
    }

    private void startSession(RemoteProfile profile) {
        checkServerCapabilities(profile);
        registerOrResumeDevice(profile);
        // 每 3-5 秒轮询（电视场景完全够用）
        schedulePoll(profile);
    }
}
```

**轮询间隔的合理性（电视场景）：**

3-5 秒轮询是正确选择，原因：
- 电视永久供电，无电池消耗问题。
- 操作频率极低，一天可能只有 1-2 次远程操作。
- 3-5 秒延迟完全可接受，适合远程切换接口、推送视频等场景。
- serverless 友好，避免 WebSocket 对 sticky session 的要求。

`RemoteAgent.start()` 行为：

1. 读取 `RemoteStore.get().profiles`。
2. 过滤出 `enabled=true`、已有设备组、或存在未消费 `pendingBindGrants` 的 profile。
3. 每个 `serverOrigin` 建一个轻量 session，互不影响；某个服务端失败不阻塞其他服务端。
4. 对每个 session 调用 `/api/server/capabilities` 做能力握手。
5. 调用 `/api/device/register` 注册或恢复在线状态；如果已有 `deviceToken/groupToken`，一并上报给 serverless 重建在线路由。
6. 如果任一 profile 开启 `keepOnline`，启动 `RemoteAgentService`；否则只在 App 进程活着时轮询。
7. 每 3-5 秒按 profile 调用 `/api/device/poll`，拿到命令后交给 `RemoteCommandExecutor`。

`RemoteAgentService` 可以直接复用 `ManageService` 的做法：

- `foregroundServiceType="dataSync|connectedDevice"`。
- 前台通知显示“远程托管在线”和服务器地址。
- 复用或抽出 `ManageService` 的电池优化引导、WakeLock、WiFiLock。
- 不启动本地网页，不把 `/m` 暴露到公网。

Manifest 需要新增：

```xml
<service
    android:name=".remote.RemoteAgentService"
    android:exported="false"
    android:foregroundServiceType="dataSync|connectedDevice"
    android:stopWithTask="false" />
```

## 6. 设备组和多设备关系

服务端和 App 都按设备组建模：

- 一个设备组里有多个设备
- 一个设备可以加入多个设备组
- 主控端 Web 控制台持有 `groupToken`（高熵 bearer token，长期有效）
- WebHTV App 也可以持有 `groupToken`，因此 App 可以作为主控端发命令，也可以作为来源设备/目标设备执行命令

基础绑定流程：

1. 被控端打开”设置 -> 增强功能 -> 远程托管”
2. 点击”生成绑定码”，App 先生成并本地保存一个 `RemoteBindGrant`，再调 `/api/device/bind-code`
3. 主控端 Web 控制台或另一台 App 输入绑定码，调 `/api/groups/claim`；如果是在已有设备组里添加设备，请求里带现有 `groupToken`，否则服务端创建新 `groupToken`。
4. 服务端返回 `groupId/groupToken/groupTokenHash/deviceId/grantId/bindGrantToken`，主控端保存到本地 `RemoteStore` 或浏览器 localStorage/IndexedDB。
5. 服务端给被控端排一个内部命令 `remote.profile.addGroup`，payload 包含 `groupId/groupToken/groupTokenHash/bindGrantToken/alias`。
6. 被控端下次 poll 收到后，先用本机 `pendingBindGrants` 校验 `bindGrantToken`，通过后把该设备组写入本机 `RemoteStore.groups`，并标记 grant 已消费。
7. 之后 serverless 实例重启也不需要重新绑定；被控端启动或轮询时会用本地保存的 `groupToken` 重建在线路由。

绑定过程有一个很小的未完成窗口：如果主控端 claim 成功后、被控端还没 poll 到 `remote.profile.addGroup`，serverless 就被回收，内部命令会丢。主控端此时已经保存了 `groupToken + bindGrantToken + targetDeviceId`，可以重发 `remote.profile.addGroup`，目标设备用本地 pending grant 接受这次补发，不需要重新生成绑定码。

`remote.profile.addGroup` 是唯一允许在目标设备尚未加入 group 前执行的 bootstrap 命令。它不能走普通 group 校验，因为目标设备还没有 `groupTokenHash`；它必须改用 `bindGrantToken` 校验。其他所有远程命令都必须要求目标设备本地存在相同 `groupTokenHash`。

添加多个目标设备：

- 每台目标设备各自生成绑定码。
- 主控端重复 claim，并在请求里带同一个 `groupToken`，同一设备组下就会出现多台设备。

添加多个主控端：

- 已有主控端显示“加入此设备组”的二维码或短码。
- 新主控端扫码后保存同一个 `groupToken`。
- 新主控端也可调用 `/api/device/register` 把自己加入设备列表，之后既能控制，也能作为一键同步来源设备。

主控端的设备列表也要本地缓存。serverless 重启后 `/api/devices` 可能暂时为空，Web 控制台应先显示本地缓存设备并标记“等待上线”；远端设备 poll 后再刷新为在线。

## 7. 一键同步抽取设计

当前打包/恢复逻辑分散在三处：

- `Action.sendBackup/getBackupBody/syncBackup`
- `Manage.syncStart/buildSyncBody`

先抽公共工具：

建议接口：

```java

                                  @Nullable BooleanSupplier running,

    public static void restore(SyncOptions options,

        public String optionsJson();
        public String backupJson();
```

2. `options.isSpider()` 时 `SyncFiles.createArchive(SyncFiles.getPaths(options.getPaths()), running, progress)`
3. `options.isLoginState()` 时 `LoginStateSync.createArchive()`
4. 计算整体 SHA256 校验和

`restore()` 内部顺序必须保持当前 `Action.syncBackup()` 行为：

1. 验证 SHA256 校验和
2. `SyncFiles.restoreArchive(syncFiles)`

## 10. serverless 简易版：零配置长期绑定

serverless 简易版只做通用在线中转，不要求 KV、R2、数据库、对象存储或必填环境变量。它必须满足一个硬约束：同一个 serverless 域名不变时，不因为实例重启、冷启动或平台回收要求用户重新绑定。

正确做法不是把所有状态放进 serverless，也不是把 `JWT_SECRET` 变成必填项，而是：

- 绑定状态保存在 App/主控端本地。
- serverless 内存只保存可重建的在线路由表、命令队列、同步临时文件。
- 设备启动、注册、轮询时把本地保存的 `deviceToken/groupToken` 带回 serverless。
- serverless 用这些 token 重建 `devices/groupDevices/online`。

### 10.1 token 设计

默认零配置模式使用高熵 bearer capability token：

```text
serverOrigin   = normalizeOrigin(request.url)
deviceToken    = dtk_<random 256-bit>
deviceId       = dev_<sha256(serverOrigin + ":" + deviceToken).slice(0, 24)>
groupToken     = gtk_<random 256-bit>
groupId        = grp_<sha256(serverOrigin + ":" + groupToken).slice(0, 24)>
groupTokenHash = sha256(serverOrigin + ":" + groupToken)
bindGrantToken = bgt_<random 256-bit>
grantId        = bnd_<sha256(serverOrigin + ":" + bindGrantToken).slice(0, 24)>
```

这些 token 不需要服务端持久存储。谁持有 `groupToken`，谁就是这个设备组的主控端；谁持有 `deviceToken`，谁就能声明自己是该设备。`deviceId/groupId/grantId` 不是独立凭证，只是由 token 和 `serverOrigin` 派生出来的稳定显示和路由 ID。

关键点：

- `deviceId` 不能再用随机值，否则 serverless 重启后无法验证“这个 deviceId 是否真的属于这个 deviceToken”。
- serverless 校验设备身份时，重新计算 `dev_<sha256(origin + ":" + deviceToken)>`，必须等于请求里的 `deviceId`。
- serverless 校验设备组时，重新计算 `groupId/groupTokenHash`，不依赖内存里是否已经存在该 group。
- 同域名、同 origin 下 token 派生结果稳定；换域名后派生 ID 变化，因此需要重新绑定。

可选增强：

- 部署者配置 `JWT_SECRET` 时，可以把 token 改成签名 token。
- 这只增强 token 可校验性，不改变核心协议。
- `JWT_SECRET` 不是简易版默认要求，不能作为第一版 P0。

### 10.2 注册和恢复在线状态

`POST /api/device/register` 同时承担“首次注册”和“实例重启后的恢复在线状态”。

首次注册：

```json
{
  "name": "Living Room TV",
  "type": 1,
  "appVersion": "x.y.z",
  "capabilities": {}
}
```

返回：

```json
{
  "ok": true,
  "deviceId": "dev_xxx",
  "deviceToken": "dtk_xxx",
  "server": {}
}
```

恢复在线状态：

```json
{
  "deviceId": "dev_xxx",
  "deviceToken": "dtk_xxx",
  "name": "Living Room TV",
  "type": 1,
  "appVersion": "x.y.z",
  "capabilities": {},
  "groups": [
    { "groupToken": "gtk_xxx" }
  ]
}
```

serverless 行为：

1. 标准化当前请求的 `serverOrigin`。
2. 首次注册没有 `deviceToken` 时，生成 `deviceToken` 并派生 `deviceId` 返回给 App。
3. 恢复注册带了 `deviceToken/deviceId` 时，重新派生 `expectedDeviceId`；不匹配直接拒绝。
4. 写入或更新内存 `devices[deviceId]` 快照。
5. 对请求里的每个 `groupToken` 计算 `groupId/groupTokenHash`，建立 `groupDevices[groupId].add(deviceId)`。
6. 返回当前能力声明和在线状态。

这样实例重启后，设备下一次启动或轮询会自动补回在线路由，不需要用户重新生成绑定码。

### 10.3 绑定流程

绑定码仍然是短期内存状态，过期或实例回收后可以重新生成；这不等于重新绑定已有设备组。

流程：

1. 被控端已经通过 `/api/device/register` 获得 `deviceId/deviceToken`。
2. 被控端生成 `bindGrantToken/grantId`，写入本机 `RemoteStore.pendingBindGrants`。
3. 被控端调用 `/api/device/bind-code` 生成 6 位短码，body 带 `deviceId/deviceToken/grantId/bindGrantToken`；serverless 校验 `deviceToken` 后，内存保存 `code -> deviceId + grantId + bindGrantToken`，10 分钟有效。
4. 主控端输入短码，调用 `/api/groups/claim`；请求里可选带现有 `groupToken`。
5. 如果请求带现有 `groupToken`，serverless 派生同一个 `groupId/groupTokenHash`；如果没带，serverless 新建 `groupToken` 并派生 ID。
6. serverless 返回 `groupId/groupToken/groupTokenHash/deviceId/grantId/bindGrantToken`，主控端保存到本地。
7. serverless 给目标设备排队 `remote.profile.addGroup` 命令。
8. 目标设备 poll 到命令后，用本地 pending grant 校验 `bindGrantToken`，通过后将 `groupId/groupToken/groupTokenHash` 写入本机 `RemoteStore.groups`。
9. 之后设备每次 register/poll 都带上本地 group 列表，serverless 由此恢复组内在线设备。

如果第 7 步命令因实例回收丢失，主控端可以基于本地保存的 `groupToken + bindGrantToken + targetDeviceId` 重发 `remote.profile.addGroup`，不需要目标设备重新生成绑定码。目标设备已经加入 group 时，该命令按幂等 upsert 处理，直接返回成功。

### 10.4 命令投递

serverless 不能把内存里的 `groupDevices` 当作长期授权来源，因为实例重启后这张表一开始为空。

命令创建规则：

- `/api/commands` 只要求主控端提供 `groupToken` 和 `targetDeviceId`。
- serverless 从 `groupToken` 计算 `groupId` 和 `groupTokenHash`，不依赖内存里的 membership 做长期授权判断。
- 如果目标设备当前在线且 serverless 已知其 group，可立即校验并入队。
- 如果目标设备暂未上线，也允许按 `targetDeviceId` 入队，等待设备 poll。
- 命令 payload 带 `groupId/groupTokenHash`。
- 被控端执行前必须检查本机 `RemoteStore.groups` 是否存在相同 `groupTokenHash`，不属于则拒绝执行。
- `remote.profile.addGroup` 例外：它是绑定 bootstrap 命令，payload 必须带 `bindGrantToken`，目标设备只检查本机 `pendingBindGrants`，成功后才写入 group。

这让实例重启后的恢复路径更顺：主控端可以先发命令，目标设备随后 poll 上线并执行；serverless 内存不是绑定关系的唯一来源。

### 10.5 能力声明

`/api/server/capabilities` 返回：

```json
{
  "ok": true,
  "serverMode": "simple",
  "serverName": "WebHTV Remote Relay",
  "relayMode": "origin-token-memory",
  "capabilities": {
    "configManage": true,
    "remoteSync": true,
    "pushAction": true,
    "recentLog": true,
    "persistentStorage": false,
    "externalObjectStorage": false,
    "deviceRevoke": false,
    "webSocket": false
  }
}
```

`origin-token-memory` 的含义：

- token 由 App/主控端按 `serverOrigin` 本地持久保存。
- serverless 只保留内存在线状态和临时任务。
- 域名不变不重新绑定。
- 域名变化视为新 server，需要重新绑定。

### 10.6 内存数据

serverless 内存保存：

```javascript
const state = {
  devices: new Map(),      // 当前实例已恢复的在线设备快照
  bindCodes: new Map(),    // 短期绑定码
  groupDevices: new Map(), // 当前实例已恢复的组内在线设备
  queues: new Map(),       // 待执行命令队列
  commands: new Map(),     // 命令执行结果
  syncs: new Map(),        // 同步任务元数据
  parts: new Map()         // 同步文件临时存储
};
```

实例回收后的影响：

- 不影响已有绑定关系；设备和主控端本地保存 token。
- `/api/devices` 初始可能为空；设备下一次 register/poll 后自动恢复在线列表。
- 未投递命令、同步任务、同步文件会丢；用户重新点击操作即可。
- 正在进行的一键同步可能失败，但不需要重新绑定设备。

### 10.7 接口契约

- `GET /api/server/capabilities`
- `POST /api/device/register` - 首次注册或恢复在线状态，返回/复用 `deviceId/deviceToken`

- `POST /api/device/poll` - 轮询命令，同时可携带本地 group 列表用于恢复在线路由
- `POST /api/groups/claim` - 加入组；body 带 `code`，可选带现有 `groupToken`
- `GET /api/devices` - 返回当前实例已恢复的在线设备视图
- `POST /api/commands` - 创建命令
- `GET /api/commands/{id}` - 查询命令状态

- `POST /api/sync/create` - 创建同步任务
- `GET /api/sync/{id}` - 查询同步状态

### 10.9 简易版限制

- 绑定不丢：同域名下不重新绑定。
- 在线状态可恢复：设备 register/poll 后重建设备列表。
- 临时任务会丢：命令队列、同步任务、同步文件仍在内存，实例回收后需要重新发起操作。
- 无强撤销：零配置模式无法可靠撤销已经泄露的 `groupToken`，需要换组或进入完整版。
- 文件大小限制：单个同步文件上限 4-25MB，按平台调整。

这些限制符合“极简部署”的定位。需要离线队列、大文件、设备撤销、备份中心时进入完整版。

## 11. 完整版 Go/Rust 服务端

完整版不是另一套 App 逻辑。它继续使用同一套 `RemoteAgent`、命令 schema、一键同步流程和 Web 控制台，只是服务端能力更强。

推荐先用 Go：

- HTTP/WebSocket/文件上传下载实现快
- 单二进制部署简单
- 数据库、对象存储、后台任务生态够用

### 11.1 部署形态

默认部署必须和 serverless 一样简单：

```bash
WEBHTV_ADDR=:8080 \
WEBHTV_DATA=./data \
WEBHTV_PUBLIC_URL=https://remote.example.com \
./webhtv-remote
```

默认值：

- `WEBHTV_ADDR=:8080`
- `WEBHTV_DATA=./data`
- SQLite：`$WEBHTV_DATA/webhtv-remote.db`
- 本地对象存储：`$WEBHTV_DATA/objects`
- Web 控制台静态文件内嵌进二进制
- 不要求 Redis、Postgres、S3、账号系统或必填 secret

可选增强：

- `WEBHTV_DB=postgres://...`：多实例或更高并发时使用。
- `WEBHTV_STORAGE=s3`、`S3_ENDPOINT/S3_BUCKET/S3_ACCESS_KEY/S3_SECRET_KEY`：大文件和长期备份使用对象存储。
- `WEBHTV_ADMIN_TOKEN`：可选服务器级管理入口，不影响普通设备组使用。

### 11.2 Go 服务端目录

```text
server/webhtv-remote/
  cmd/webhtv-remote/main.go
  internal/config/          # 环境变量、默认路径、public URL
  internal/httpapi/         # REST API、CORS、静态 Web 控制台
  internal/ws/              # WebSocket 设备通道
  internal/relay/           # register、poll、commands、sync 状态机
  internal/store/           # SQLite/Postgres DAO
  internal/storage/         # local/S3 对象存储
  internal/jobs/            # TTL 清理、快照保留策略
  internal/model/           # DTO 和 DB model
  web/console/              # 与 serverless 共用的 Web 控制台源码
```

Rust 也可以做，但第一版不建议。Go 的 HTTP、WebSocket、SQLite、文件上传、单二进制分发会更快完成。

### 11.3 完整版能力边界

完整版必须覆盖简易版全部能力，并新增：

- **WebSocket 长连接**：设备在线时命令即时下发；断开后自动退回 3-5 秒轮询。
- **持久命令队列**：设备离线时命令不丢，直到 TTL 到期或用户取消。
- **持久同步任务**：同步任务元数据、part、状态可恢复。
- **对象存储**：保存同步 part、备份快照和调试包，默认本地磁盘，可选 S3/MinIO。
- **设备管理**：重命名设备、移除设备、清理离线设备、查看在线状态。
- **备份中心**：一键同步完成后可保存快照，支持选择快照恢复到任意目标设备。
- **完整 App 能力远程操作**：壳代理、站点注入、WebHome 扩展、文件管理、批量设备。
- **历史记录**：命令、同步、快照、设备上线记录。

### 11.6 WebSocket 和轮询兼容

App 端逻辑不分叉：

1. `RemoteAgent` 先调用 `/api/server/capabilities`。
2. 如果 `capabilities.webSocket=true`，尝试连接 `/api/device/ws`。
3. WebSocket 连接成功后，命令由服务端 push。
4. WebSocket 断开、失败或系统限制后台连接时，自动回退 `/api/device/poll`。
5. 轮询协议和 serverless 完全一致。

WebSocket 首包：

```json
{
  "type": "hello",
  "deviceId": "dev_xxx",
  "deviceToken": "dtk_xxx",
  "groups": [{ "groupToken": "gtk_xxx" }],
  "capabilities": {}
}
```

服务端返回：

```json
{
  "type": "ready",
  "serverMode": "full",
  "capabilities": {}
}
```

命令 push：

```json
{
  "type": "command",
  "command": {}
}
```

执行结果：

```json
{
  "type": "result",
  "commandId": "cmd_xxx",
  "ok": true,
  "message": "",
  "data": {}
}
```

### 11.10 简易版迁移到完整版

推荐一开始就使用自定义域名，例如：

```text
https://remote.example.com
```

先指向 Cloudflare Worker/Deno/Vercel，后续再把同一个域名切到 Go 完整版。因为 `deviceId/groupId/grantId` 都由 `serverOrigin + token` 派生，只要 `serverOrigin` 不变，就不需要重新绑定。

迁移流程：

1. 部署 Go 完整版。
2. 把原 serverless 域名的 DNS/CNAME 切到完整版。
3. App 下一次 register/poll 时用本地 `deviceToken/groupToken` 恢复设备和设备组。
4. 完整版把恢复到的 devices/groups/memberships 写入数据库。
5. Web 控制台读取本地缓存的 `groupToken`，刷新设备列表。

如果换了域名，`serverOrigin` 变化，派生 ID 会变化，需要重新绑定。这不是服务端限制，而是当前“同域免重绑”原则的边界。

## 12. Web 控制台设计

Web 控制台只有一个工程，按服务端握手结果显示不同界面：

- `serverMode=simple`：设备列表、接口管理、一键同步、推送/搜索、最近日志
- `serverMode=full`：在简易版基础上显示设备管理、备份中心、文件管理、壳代理、站点注入、WebHome 扩展、批量设备、完整诊断

主控端同步操作只给三个主动作：

1. 选择来源设备。
2. 选择目标设备。
3. 勾选数据项并点击同步。

WebHome 默认勾选。接口相关 Cookie、Token、缓存文件由一键同步内部处理，不提供通用 Cookie 输入框，不要求用户导出、选择或上传任何文件。

Web 控制台必须本地保存 `groupToken` 和设备缓存。serverless 重启后 `/api/devices` 可能暂时为空，控制台不能因此提示重新绑定；应该显示缓存设备并标记“等待上线”，等设备下一次 poll/register 后恢复在线状态。

完整版 Web 控制台页面：

| 页面 | simple | full | 说明 |
| --- | --- | --- | --- |
| 设备 | 是 | 是 | 设备列表、在线状态、设备能力、最近命令、重命名 |
| 一键同步 | 是 | 是 | 来源设备、目标设备、数据项、进度；full 可保存为快照 |
| 接口管理 | 是 | 是 | 配置列表、添加、切换、删除 |
| 推送/搜索 | 是 | 是 | 远程搜索、推送播放 |
| 日志 | 是 | 是 | 最近关键日志 |
| 备份中心 | 否 | 是 | 快照列表、恢复、删除、保留时间 |
| 文件管理 | 否 | 是 | 受限目录浏览、文本预览、文本写入 |
| 壳代理 | 否 | 是 | 开关、默认代理地址、规则编辑、站点建议 |
| 站点注入 | 否 | 是 | `CustomCspSetting.Registry` UI/JSON 双模式、页面代码保存 |
| WebHome 扩展 | 否 | 是 | 用户源列表、保存、启停、删除、reload、debug report |

## 14. 实施顺序

### 第一阶段：serverless 长期绑定改造（P0 必须）

目标：同一个 serverless 域名不变时，实例重启后不重新绑定。

- 将当前 `deviceSecret/groupToken` 命名和协议统一为 `deviceToken/groupToken`。
- `deviceId/groupId/grantId` 改为由 `serverOrigin + token` 派生，不再使用随机 `deviceId`。
- `/api/device/register` 支持首次注册和恢复在线状态。
- App 已有 `deviceId/deviceToken/groups` 时，register body 必须带回本地保存的 group 列表。
- serverless 如果内存没有该设备，重新派生 `deviceId` 校验通过后恢复设备快照。
- `/api/device/poll` 支持携带 group 列表，或在 401/unknown device 时让 App 自动 register 后重试。
- `/api/device/bind-code` 支持 `grantId/bindGrantToken`，用于 claim 后命令丢失时补发 `remote.profile.addGroup`。
- `/api/groups/claim` 支持可选现有 `groupToken`，返回 `groupId/groupToken/groupTokenHash/deviceId/grantId/bindGrantToken`，并给目标设备排 `remote.profile.addGroup`。
- `/api/commands` 不再硬依赖当前内存 `groupDevices` 才能入队；目标设备执行前做本地 group 校验。
- `remote.profile.addGroup` 作为 bootstrap 命令，只走 `bindGrantToken` 校验；普通命令走 `groupTokenHash` 校验。
- Web 控制台和 App 主控端保存设备缓存，serverless 重启后先显示“等待上线”。

## 16. 关键设计决策总结

### 保持的设计

1. **3-5 秒轮询**
   - 电视永久供电，无电池消耗
   - 操作频率低，3-5 秒延迟完全可接受
   - serverless 友好，部署简单

2. **长期有效令牌**
   - 电视固定设备，长期使用
   - 电视无键盘，频繁认证体验差
   - 同域名下不重新绑定
   - App 令牌保存到外部文件，卸载重装后仍可恢复
   - 参考 Home Assistant、Roku TV、Apple TV 的长期凭证思路

3. **简化安全机制**
   - HTTPS + 高熵 bearer token
   - 不需要账号系统、细粒度权限、端到端加密
   - `JWT_SECRET` 是可选增强，不是简易版必填项

4. **设备对等架构**
   - 借鉴 Syncthing/KDE Connect
   - 任意设备可主控、可被控、可同步
   - 不固定”控制版/被控版”

5. **命令模式**
   - 结构化 command（如 `config.use`）
   - 不代理原始 HTTP `/manage/*`
   - 参考 Home Assistant 服务动作

### 修正后的 serverless 边界

1. **长期绑定不依赖内存**
   - `deviceToken/groupToken` 保存在 App/主控端本地
   - App 端主存储是 `WebHTV/RemoteTrust/profile.json` 外部文件，Prefers 只是镜像
   - serverless 重启后由设备 register/poll 重建在线路由
   - 域名不变不重新绑定，域名变化重新绑定

2. **临时任务仍依赖内存**
   - 未投递命令、同步任务、同步文件在实例回收后会丢
   - 用户重新发起操作即可，不影响绑定关系

3. **TTL 调整**
   - 命令：30分钟 → 1小时
   - 同步：30分钟 → 2小时
   - 绑定码：10分钟（不变）

4. **SHA256 校验**
   - 防止传输损坏
   - 不是安全加密，是完整性校验

## 17. 结论

推荐第一版就叫”远程托管”。核心不是远程桌面，也不是复杂账号系统，而是：

1. 设备组里添加多台 WebHTV App。
2. 任意 App 都可主控、可被控、可作为来源、可作为目标。
3. 主控端通过 Web 控制台发结构化命令。
4. 被控端主动连接中转服务并执行现有 App 能力。
5. 一键同步是主通道，WebHome 默认包含，接口相关登录文件作为同步内部数据自动处理。
6. serverless 同域名下不重新绑定；实例重启后由 App/主控端本地 token 恢复在线状态。
7. 简易版保持零配置，完整版解决持久队列、设备撤销、大文件和备份中心。

当前 P0 任务有两个：

1. 把 serverless 从纯 `online-memory` 改成 `origin-token-memory`：`deviceId/groupId/grantId` 由 `serverOrigin + token` 派生，支持 `bindGrantToken` bootstrap 补发。
2. 把 App 端 `RemoteStore` 做成外部文件主存储：卸载重装后恢复 profile，并从普通一键同步/备份里排除远程托管令牌。
